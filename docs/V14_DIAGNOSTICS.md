# V14 Diagnostics — Runtime Cost + Collapse Forensics

Local instrumentation evidence, not official Kaggle scores. Generated by
`scripts/run_v14_diagnostics.py` (instrumentation and analysis only; no
production default behavior was changed). Raw data:
`submissions/v14_diagnostics.json`.

- Bound: `--max-timepoints 12`, samples `44b6_0113de3b`, `44b6_0b24845f`, `44b6_0c582fdc`.
- CFAR calibration matches the recorded V14 experiments: training radius `(1,6,6)`,
  guard `(0,1,1)`, sidelobe radius `(1,12,12)`, detection cap `900`.
- Stage timings measure the detection path only (no linking), so they isolate
  where the CFAR+sidelobe route adds cost over the adaptive baseline.

## Part 1 — Where hybrid runtime cost concentrates

Mean per-sample stage time over the three ablation runs (12 timepoints each):

| stage | mean ms | % of hybrid detection time |
|---|---:|---:|
| cfar_background_stats | ~9,600 | ~59–62% |
| peak_local_max | ~4,900 | ~29–31% |
| sidelobe_suppression | ~400–1,460 | ~2.5–8.7% |
| normalize | ~510 | ~3% |
| cfar_threshold_select | ~274 | ~1.7% |
| detection_build | ~20 | ~0.1% |

Reference: mean adaptive **baseline total** (detection + linking) was **~5,342 ms**;
mean **hybrid detection-only** total was **~16,200–16,350 ms**, a mean delta of
**~10,850–11,010 ms**.

**Finding:** the extra cost is **not spread evenly** — it is concentrated in a
single stage. `cfar_background_stats` (the box-ring mean/variance estimate,
implemented as four `scipy.ndimage.uniform_filter` passes over the full 3D
volume) accounts for ~60% of hybrid runtime by itself and is pure additive
overhead the baseline never pays. The `peak_local_max` maximum filter is the
second contributor (~30%), but the baseline local-maxima detector already pays a
comparable peak-filter cost, so the *net* added cost is dominated by
`cfar_background_stats`.

**Implication:** if hybrid is ever revisited, optimize `cfar_background_stats`
first (e.g. separable/integral-image summed-area tables instead of four full
`uniform_filter` passes, or single-pass mean+meansq). Tuning detection
parameters will not move runtime.

## Part 2 — Which parameter drove the result (controlled ablation)

Two parameters were changed together in the earlier sigma/isotropic test. Here
they are isolated (sigma at 0.6 vs baseline 1.1; isotropic floor at 0.7 vs
baseline 0.85):

| run | cfar_k_sigma | sidelobe_floor | mean hybrid ms | mean delta vs baseline ms |
|---|---:|---:|---:|---:|
| A — sigma 0.6 only | 0.6 | 0.85 | 16,353 | 11,012 |
| B — iso floor 0.7 only | 1.1 | 0.70 | 16,202 | 10,860 |
| C — both | 0.6 | 0.70 | 16,185 | 10,844 |

**Finding:** runtime is **essentially identical across A, B, and C** (within
~1%). Neither the sigma threshold nor the isotropic floor is responsible for the
cost — the cost is **structural** (the CFAR background estimate + peak filter run
regardless of parameter values). Kept-detection counts also barely move between
runs, and prior sparse-recall results were unchanged versus baseline. So:

- Neither parameter is responsible for the **runtime** — the CFAR machinery is.
- Neither parameter produced a **quality** gain over baseline in the recorded runs.

The two-parameter change was a confound with no measurable upside; there is no
single parameter worth continuing to tune on the current sigma/isotropic path.

## Part 3 — Collapse forensics for `44b6_0c582fdc` (pfa/axial)

### Sample characterization (12-timepoint medians)

| sample | fg fraction | largest comp voxels | baseline peak count | background mean (norm) | background std (norm) | SNR proxy | routes to CFAR? |
|---|---:|---:|---:|---:|---:|---:|:--:|
| 44b6_0c582fdc (collapse) | 0.084 | 208,592 | 401 | **0.218** | 0.150 | 3.15 | **yes** |
| 44b6_0113de3b | 0.020 | 2,378 | 168 | 0.084 | 0.080 | 7.56 | no (clean/components) |
| 44b6_0b24845f | 0.232 | 950,582 | 514 | 0.319 | 0.192 | 2.05 | no (>600k / >0.20 gate) |

Only `44b6_0c582fdc` satisfies the `merged_6bba_only` routing gate
(largest component 100k–600k **and** foreground fraction 0.05–0.20), so it is the
only one of the three that actually enters the CFAR path — which is why it is the
only one that can collapse.

### Root cause: bounded signal vs multiplicative CA-CFAR threshold

The pfa mode uses a CA-CFAR multiplicative threshold `T = alpha * background_mean`
with `alpha = N * (pfa^(-1/N) - 1)`, `N = ring_count = 498` for training `(1,6,6)`
minus guard `(0,1,1)`. The signal is robust-normalized to `[0, 1]`, so any voxel
whose adaptive threshold `T` exceeds ~1.0 can **never** be exceeded, and every
peak is rejected.

Computed thresholds on the normalized scale:

| pfa | alpha | T at bg=0.218 (collapse sample) | T at bg=0.084 (clean sample) |
|---:|---:|---:|---:|
| 1e-3 | 6.96 | **1.518** (> 1 → collapse) | 0.585 |
| 1e-2 | 4.63 | **1.010** (> 1 → collapse) | 0.389 |
| 2e-2 | 3.93 | 0.857 (recovers) | 0.330 |
| 5e-2 | 3.00 | 0.656 | 0.253 |
| 1e-1 | 2.31 | 0.504 | 0.194 |

Measured sweep (pfa/axial, collapse sample): `pfa=1e-3` and `pfa=1e-2` both gave
**`pre_sidelobe_detection_count = 0`** — i.e. everything was killed at the CFAR
threshold stage, **before** any sidelobe suppression ran.

**Findings:**

1. The collapse is a **CFAR pfa-threshold** failure, **not** an axial sidelobe
   failure. `pre_sidelobe_detection_count = 0` proves suppression never even ran;
   the axial mode is exonerated for this collapse.
2. The trigger is **data-dependent**: `44b6_0c582fdc` has a relatively high
   normalized background mean (~0.218). Combined with an aggressive pfa
   (`≤ 1e-2`, i.e. `alpha ≥ 4.6`), the multiplicative threshold `alpha * mean`
   lands **above the normalized ceiling of 1.0**, so zero peaks survive. The
   clean sample (bg ~0.084) stays well under 1.0 at the same pfa and never
   collapses.
3. **Threshold sensitivity bound:** for this sample the collapse boundary is
   `pfa ≈ 0.015–0.02`. `pfa = 1e-2` sits right on the edge (T = 1.010) and still
   collapses; `pfa ≥ 0.02` (T ≤ 0.857) is predicted to recover detections. The
   two swept values (`1e-3`, `1e-2`) were both inside the collapse region, so
   `first_non_collapsing_pfa` came back `null` — as expected from the math, not a
   sign the failure is unbounded.

### Structural conclusion

The pfa CA-CFAR formulation assumes **unbounded, exponentially-distributed
clutter**, but the pipeline feeds it a signal clamped to `[0, 1]`. That mismatch
is the real defect: for any sample whose local background mean is high enough,
`alpha * mean` can exceed the signal ceiling and annihilate all detections. This
is why pfa+axial remains correctly parked behind `--allow-unsafe-pfa-axial`. If
pfa mode is ever revived, the fix is at the formulation level (e.g. cap `T` below
the achievable max, add an additive/log-domain floor, or apply CFAR on an
unnormalized/less-clamped intensity), not merely picking a larger pfa per sample.

## Summary

- **Runtime cost** concentrates in `cfar_background_stats` (~60% of hybrid time,
  ~9.6 s/12 tp), the single dominant additive overhead over baseline.
- **Neither** sigma nor the isotropic floor drives the runtime or the (absent)
  quality change; the cost is structural to the CFAR machinery.
- **`44b6_0c582fdc` collapse** is caused by its high normalized background mean
  (~0.218) interacting with the multiplicative CA-CFAR threshold: at `pfa ≤ 1e-2`
  the threshold exceeds the `[0,1]` signal ceiling and rejects every peak. It is
  predicted to stop collapsing at `pfa ≥ ~0.02`. The collapse originates in CFAR
  thresholding, not axial sidelobe suppression.
- **No production/default changes** were made in this pass.
